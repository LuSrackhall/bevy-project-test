## Context

当前 `arrow_movement_system` 每 Tick 收集 `SoldierMarker` 实体进行碰撞检测，城池实体（`CityMarker`）完全不在碰撞查询范围内。箭矢飞行阶段仅对兵种造成伤害，城池建筑对箭矢透明。

城池实体使用独立的生命值组件 `CityComponent { health_current, health_max, ... }`（不同于兵种的 `Health { current, max }`），且已有独立的"士兵自杀式攻城"机制（`city_interaction_system`）来摧毁城池。城池同时拥有 `CityRadius(pub u32)` 组件表示其视觉/交互半径。

## Goals / Non-Goals

**Goals:**
- 箭矢飞行阶段与城池建筑发生碰撞检测
- 建筑碰撞伤害以 1/200 比例计算，通过整数累加器保证确定性和精度
- 建筑碰撞后箭矢强制进入衰减（不穿透）
- 穿透兵种后的箭矢在同一 Tick 继续检查建筑碰撞
- 己方箭矢穿过己方城池

**Non-Goals:**
- 不改变现有自杀式攻城机制
- 不改变兵种碰撞检测逻辑
- 不改变箭矢生成逻辑
- 不完全阻止箭矢对城池的远程伤害（这是兵种自杀式攻城的职责，本改动仅添加微量的箭矢溅射效果）

## Decisions

### D1: 使用 u32 整数累加器（而非 Fixed 定点数）

**选择**: `CityComponent` 新增 `arrow_damage_acc: u32`，每次命中累加 `arrow.damage`，满 200 时扣 1 点城池血量。

**理由**: 城池生命值是 `u32`，伤害比例为 1/200。用 `u32 分子 / 200` 的整数累加天然匹配精度需求，且比 `Fixed` 定点数运算更简单。13 支 16 攻的箭（16×13=208）扣 1 血，余 8 留存。全整数运算，完全确定性。

**替代方案**: 用 `Fixed` 累加。开销相同，但 `Fixed` 在取整时需要额外转换步骤，不增加价值。

### D2: 城池碰撞检测放在兵种碰撞之后（同一 Tick）

**选择**: 箭矢在同一 Tick 内先完成所有兵种碰撞检测，若穿透兵种后未被停止，再检查城池碰撞。

**理由**: 逻辑清晰——箭矢穿透前排兵种后射中后方城墙，符合物理直觉。兵种碰撞检测使用 `ARROW_HIT_RADIUS = 22` 单位，足够大以防止穿透后"跳过"城池。城墙碰撞优先于兵种（把城池放前面）反而会导致箭矢大量被城墙截断，削弱弓兵的对兵种压制能力。

### D3: 建筑碰撞后强制衰减（不判断穿透）

**选择**: 命中建筑后直接设置 `decay_remaining = ARROW_DECAY_TICKS`，箭矢停留在命中位置或跟随城池（如城池有移动）。

**理由**: 建筑不是兵种，穿透建筑在概念上不合理（建筑是实心结构）。此外 `pierce_chance` 是针对兵种设计的平衡参数，不应套用到建筑。

## Risks / Trade-offs

- **[城池血量极缓慢下降]**：箭矢 1/200 伤害极低，玩家几乎感知不到。这对 design 来说是符合预期的——城池破坏的主要手段是自杀式攻城，箭矢只是"视觉和逻辑"上的物理交互。→ 已接受。
- **[城池成为箭矢"海绵"]**：大规模团战时，大量箭矢可能被敌对城池吸收，削弱弓兵对后排兵种的威胁。这实际上是合理的策略效果——城池提供物理掩护。→ 已接受。
- **[CityComponent 字段膨胀]**：新增 4 字节。→ 可忽略。
